Context

Started with nothing. Built something worth writing about.

When this work began, there was no enterprise data foundation. Raw data sat in silos — network telemetry, operational systems, customer data — with no unified layer for analytics, reporting, or eventually AI.

The decision to build on Snowflake was deliberate: a cloud-native warehouse with strong multi-cloud support, mature governance, and the ability to scale without infrastructure management overhead. The platform was built from scratch — schema design, ingestion pipelines, governance frameworks, analytics layers — and scaled to serve real-time intelligence across 300+ global venues.

It worked well enough that Snowflake featured the work in an official customer case study on snowflake.com — recognizing the S.M.A.R.T. network analytics and operational intelligence architecture.

The Decision

Why migrate a platform that was working?

This is the question that matters. The Snowflake platform was functioning well. Moving it carried real risk — data integrity, pipeline continuity, team retraining, cost. The case for migration had to be stronger than "GCP is newer."

Snowflake Platform
Strong analytics layer — warehouse-first architecture optimized for SQL-based querying
Multi-cloud capable — ran across AWS and GCP with data sharing features
Separate cost model — compute and storage pricing outside the primary cloud spend
Separate ecosystem — integrations with GCP AI/ML stack required additional plumbing
BigQuery Platform
Native GCP integration — directly connected to Vertex AI, Agentspace, Pub/Sub, and the full agentic infrastructure
Agent-native access — MCP layer on BigQuery enables AI agents to query live data without cross-cloud hops
Unified cost model — consolidated cloud spend with GCP committed use discounts applied across the stack
Serverless at scale — no cluster management, cost-per-query model aligned with variable workloads
How It Was Done

Phases of migration

01
Foundation
Build the Snowflake platform to production maturity

Designed and built the full data lake on Snowflake — ingestion pipelines, transformation layers, governance framework, analytics surfaces. Scaled to 10PB+ serving real-time intelligence across 300+ venues. Earned the Snowflake case study at this stage.

02
Decision
Make the architectural case for GCP consolidation

As the agentic AI strategy matured, the friction of operating a separate data platform outside the primary GCP ecosystem became the constraint. The decision was made to consolidate onto BigQuery — not because Snowflake failed, but because the future architecture required native AI integration.

03
Migration
Migrate pipelines, schemas, and governance to BigQuery

Rebuilt ingestion pipelines, translated schema designs, migrated transformation logic, and re-established governance controls on BigQuery. Maintained production continuity throughout — no data gaps, no analytics downtime.

04
Integration
Build the MCP layer — enabling agent-native data access

With BigQuery as the production intelligence layer, built the MCP interface on top — enabling AI agents across the enterprise to query live production data directly as part of their decision loops. This was the architecture the migration was designed to enable.

Key Decisions

What the migration required

Timing
Migrate after proving the platform, not before. The Snowflake platform was built to production maturity first — including the case study. Migration happened when the architectural reason was clear, not because of platform preference.
Continuity
No analytics downtime as a hard requirement. 300+ venues and hundreds of internal stakeholders depended on continuous data availability. The migration was phased to maintain production SLAs throughout.
Rationale
The migration unlocked the agentic architecture. The primary reason to consolidate onto BigQuery was to enable native integration with the GCP AI stack — Vertex AI, Agentspace, ADK — and to build the MCP layer that made agent-native data access possible.
